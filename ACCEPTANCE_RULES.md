# ACCEPTANCE_RULES

## Что НЕЛЬЗЯ считать выполнением
- «Код выглядит корректно» без запуска проверок.
- Только code-review/статический аудит без runtime evidence для сценариев, требующих выполнения.
- PASS в Linux для macOS-only сценариев (Preview/Print/AppKit dialogs/packaging).
- Локальный успех одного скрипта как доказательство полной release-ready готовности.
- Утверждения «должно работать на Mac», если не приложены реальные macOS результаты.

## Обязательный evidence pack
Для любой задачи, претендующей на закрытие:
1. **Команды проверки**: точные команды + exit code + дата/время.
2. **Артефакты выполнения**:
   - для core: вывод тестов/скриптов;
   - для UI/macOS: скриншоты/видео + экспортированные файлы (если релевантно).
3. **Обновление статусов документов**:
   - `CURRENT_STATE.md`;
   - `DEFECT_BACKLOG.md` (если затронуто);
   - при необходимости `ACCEPTANCE_CHECKLIST.md`.
4. **Traceability**: какой именно acceptance criterion закрыт каким evidence.

## Когда merge запрещён
Merge запрещён, если выполняется хотя бы одно:
- Поставлен PASS без воспроизводимой проверки.
- Изменение заявлено как release-ready, но остаются P0 дефекты из backlog.
- Для затронутого сценария нет evidence pack.
- Статусы документов противоречат фактическим результатам команд.
- По задаче есть `BLOCKED_ENV`, но это скрыто/не отражено в статусах.

## Что такое PASS / FAIL
- **PASS**: критерий подтверждён фактическим запуском в релевантной среде, evidence сохранён, противоречий нет.
- **FAIL**: критерий не выполнен, либо не подтверждён в релевантной среде, либо есть явная ошибка/красный результат.
- **UNCONFIRMED** (допустимый промежуточный статус): данных недостаточно для PASS/FAIL; merge-решение принимается как для FAIL до появления evidence.
